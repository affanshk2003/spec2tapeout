## Overview

The pipeline accepts a chip design specification and a behavioral Verilog testbench and produces:

1. **Functionally verified RTL** — generated by a local LLM, verified by Iverilog simulation
2. **Gate-level netlist** — synthesized by Yosys targeting the Sky130HD standard cell library
3. **Post-synthesis testbench** — LLM-converted and verified against the gate-level netlist
4. **SDC timing constraints** — extracted from the specification
5. **ODB databases** — one per P&R stage (floorplan → place → CTS → route → final)
6. **GDS layout** — DRC-clean, ready for fabrication sign-off

## Pipeline Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        pipeline.py                              │
│              (single-command orchestrator)                       │
└───────┬─────────────────┬──────────────────┬────────────────────┘
        │                 │                  │
        ▼                 ▼                  ▼
┌───────────────┐ ┌───────────────┐ ┌───────────────────────────┐
│  agent_v1_5   │ │  agent_v2_4   │ │       agent_v3_3          │
│               │ │               │ │                           │
│ Spec → RTL    │ │ RTL → Netlist │ │    Netlist → GDS          │
│               │ │               │ │                           │
│ DeepSeek-     │ │ Yosys         │ │ OpenROAD / ORFS           │
│ Coder 6.7B    │ │ DeepSeek-     │ │ (make all)                │
│ + Iverilog    │ │ Coder 6.7B    │ │                           │
│               │ │ + Iverilog    │ │ 6 stages:                 │
│ Up to 50      │ │               │ │ synth → floorplan →       │
│ attempts      │ │ Up to 10 TB   │ │ place → CTS →             │
│               │ │ attempts      │ │ route → final             │
└───────────────┘ └───────────────┘ └───────────────────────────┘
        │                 │                  │
        ▼                 ▼                  ▼
   rtl/*.v        synthesized/*.v      results/.../
                  constraints/*.sdc    6_final.odb
                  config/*.mk          6_final.gds
```

## Environment Setup

The pipeline runs across **two Docker containers** — one for RTL/synthesis work and one for ORFS/P&R. Ollama runs on your **Windows host** and is accessed from inside Docker via `host.docker.internal`.

### Prerequisites

| Tool | Where it runs | Download |
|---|---|---|
| Docker Desktop | Windows host | https://www.docker.com/products/docker-desktop |
| Ollama | Windows host | https://ollama.com |
| DeepSeek-Coder 6.7B | Windows host (via Ollama) | `ollama pull deepseek-coder:6.7b` |
| VcXsrv (for GUI) | Windows host | https://sourceforge.net/projects/vcxsrv |

### Ollama LLM Server Setup (Windows Host)

Ollama must be running on your Windows host **before** starting any agent. Open a dedicated terminal:

```cmd
set OLLAMA_HOST=0.0.0.0:11434
ollama serve
```

In a **second** terminal, pull the model (one-time setup):

```cmd
ollama pull deepseek-coder:6.7b
```

Verify the model is available:

```cmd
ollama list
# Should show: deepseek-coder:6.7b
```

> **Important:** `OLLAMA_HOST=0.0.0.0:11434` binds Ollama to all interfaces,
> allowing the Docker container to reach it via `host.docker.internal:11434`.
> Without this, Docker containers cannot connect to Ollama.

### Connecting Docker to Ollama

Inside either Docker container, verify connectivity to Ollama on the host:

```bash
curl http://host.docker.internal:11434/api/tags
# Expected: JSON response listing available models including deepseek-coder:6.7b
```

If this fails:
- Confirm `ollama serve` is running on the host with `OLLAMA_HOST=0.0.0.0:11434`
- Confirm the container was started with `--add-host host.docker.internal:host-gateway`
- Check Windows Firewall is not blocking port 11434

The agent scripts use:
```python
OLLAMA_URL = "http://host.docker.internal:11434/api/generate"  # from Docker
# or
OLLAMA_URL = "http://localhost:11434/api/generate"             # if running natively
```

## Running the Full Pipeline (Single Command)

Run everything — RTL generation, synthesis, place and route — with one command.

**In the ORFS Docker container** (which has all tools available):

```bash
cd /workspace

python3 pipeline.py \
  --spec problems/p1.yaml \
  --tb testbench/p1_tb.v \
  --cells ./skywater-pdk-libs-sky130_fd_sc_hd/cells \
  --orfs /OpenROAD-flow-scripts/flow
```

With optional GUI launch after P&R:

```bash
python3 pipeline.py \
  --spec problems/p1.yaml \
  --tb testbench/p1_tb.v \
  --cells ./skywater-pdk-libs-sky130_fd_sc_hd/cells \
  --orfs /OpenROAD-flow-scripts/flow \
  --open-gui
```

Override the ORFS platform manually (default: auto-detected as `sky130hd`):

```bash
python3 pipeline.py \
  --spec problems/p1.yaml \
  --tb testbench/p1_tb.v \
  --cells ./skywater-pdk-libs-sky130_fd_sc_hd/cells \
  --orfs /OpenROAD-flow-scripts/flow \
  --platform sky130hd
```

**What happens:**

| Phase | What pipeline.py does |
|---|---|
| Phase 1 | Runs Agent V1 → generates and verifies RTL → saves to `rtl/` |
| Phase 2 | Runs Agent V2 → synthesizes RTL → saves netlist, SDC, config.mk |
| Phase 3 | Stages files into ORFS design tree, normalizes config.mk |
| Phase 4 | Runs Agent V3 → full ORFS P&R → ODB + GDS in `results/` |
| Optional | Launches OpenROAD GUI with final ODB loaded |

## Running Agents Individually

### Agent V1 — RTL Generation

**Container:** Ubuntu (Yosys/Iverilog) or ORFS
**Requires:** Ollama running on host

```bash
cd /workspace

python3 agent_v1_5.py \
  --spec problems/p1.yaml \
  --tb testbench/p1_tb.v
```

Optional arguments:
```
--rtl-out rtl/custom_name.v    Override output RTL filename
--max-attempts 50              Max generation attempts (default: 50)
--model deepseek-coder:6.7b    LLM model name
```

**Output:** `rtl/<module_name>.v` — functionally verified synthesizable Verilog RTL

---

### Agent V2 — Synthesis

**Container:** Ubuntu (Yosys/Iverilog) or ORFS
**Requires:** Verified RTL from Agent V1; Ollama running on host; Sky130HD cells directory

```bash
cd /workspace

python3 agent_v2_4.py \
  --spec problems/p1.yaml \
  --rtl rtl/seq_detector_0011.v \
  --tb testbench/p1_tb.v \
  --cells ./skywater-pdk-libs-sky130_fd_sc_hd/cells
```

**Outputs:**

| File | Description |
|---|---|
| `synthesized/<module>_synth.v` | Gate-level netlist (Sky130HD cells) |
| `constraints/<module>.sdc` | SDC timing constraints |
| `config/<module>_config.mk` | ORFS configuration file |
| `testbench/<module>_tb_postsynthesis.v` | Post-synthesis testbench |
| `logs/<module>_synth.log` | Yosys synthesis log |
| `logs/<module>_verify_postsynthesis.log` | Post-synthesis simulation log |

### Agent V3 — Place and Route

**Container:** ORFS only (`openroad/orfs`)
**Requires:** Synthesized netlist, SDC, and config.mk from Agent V2

```bash
cd /workspace

python3 agent_v3_3.py \
  --netlist synthesized/seq_detector_0011_synth.v \
  --sdc constraints/seq_detector_0011.sdc \
  --config config/seq_detector_0011_config.mk \
  --orfs /OpenROAD-flow-scripts/flow
```

Optional arguments:
```
--target all                   ORFS make target (default: all)
--skip-cts-repair-timing       Skip CTS repair timing step (default: on)
--no-skip-cts-repair-timing    Enable CTS repair timing step
```

**Outputs (inside ORFS results directory):**

| File | Description |
|---|---|
| `results/sky130hd/<design>/base/1_synth.odb` | After synthesis |
| `results/sky130hd/<design>/base/2_floorplan.odb` | After floorplan |
| `results/sky130hd/<design>/base/3_place.odb` | After placement |
| `results/sky130hd/<design>/base/4_cts.odb` | After clock tree synthesis |
| `results/sky130hd/<design>/base/5_route.odb` | After routing |
| `results/sky130hd/<design>/base/6_final.odb` | Final signed-off database |
| `results/sky130hd/<design>/base/6_final.gds` | Final GDS layout |
| `<design>_pnr_report.txt` | P&R summary report |

## Input/Output Description

### Inputs

| Input | Format | Description |
|---|---|---|
| Design specification | YAML or plain text | Describes module name, ports, behavior, timing |
| Behavioral testbench | Verilog (`.v`) | Functional testbench that outputs PASS/FAIL |
| Cell library | Directory of `.v` files | Sky130HD standard cell Verilog models |

**Specification format example (`problems/p1.yaml`):**
```yaml
seq_detector_0011
module: seq_detector_0011
description: >
  Moore FSM sequence detector for pattern 0011.
  Outputs detected=1 one clock cycle after the complete
  sequence 0→0→1→1 is observed on data_in.
ports:
  inputs: [clk, reset, data_in]
  outputs: [detected]
clock: clk
reset: reset
clock_period_ns: 20.0
```

### Outputs

All outputs are written to the workspace directory:

```
rtl/                          ← Agent V1 output
synthesized/                  ← Agent V2 output
constraints/                  ← Agent V2 output
config/                       ← Agent V2 output
testbench/*_postsynthesis.v   ← Agent V2 output
logs/                         ← All agents write logs here
/OpenROAD-flow-scripts/flow/results/sky130hd/<design>/base/
                              ← Agent V3 output (ODB + GDS)
```

## How to Run Hidden Testcases

The grader can reproduce results on any hidden testcase by:

1. **Placing the spec file** in `problems/<testcase>.yaml`
2. **Placing the testbench** in `testbench/<testcase>_tb.v`
3. **Running the pipeline:**

```bash
python3 pipeline.py \
  --spec problems/<testcase>.yaml \
  --tb testbench/<testcase>_tb.v \
  --cells ./skywater-pdk-libs-sky130_fd_sc_hd/cells \
  --orfs /OpenROAD-flow-scripts/flow
```

The pipeline auto-detects the module name from the RTL, auto-stages all files, and produces outputs in the standard directories. No other configuration is needed.

**Requirements for a hidden testcase to work:**
- The YAML spec must have the module name on the first line (or under `module:` key)
- The testbench must print `PASS` on success and `FAIL` on failure
- The design must be synthesizable with Sky130HD standard cells
- The design must be compatible with a 20 ns clock period (adjustable in the spec)

## Known Issues

| Issue | Status | Workaround |
|---|---|---|
| Agent V2 generates incorrect ORFS variable names in config.mk (PROCESS/PDK instead of PLATFORM) | Known — pipeline.py fixes this automatically at runtime | None needed when using pipeline.py |
| SystemVerilog constructs in LLM-generated RTL cause Iverilog post-synthesis simulation failures | Active — workaround is UDP cell filter | Agent V2 skips 54 UDP-dependent cell types |
| OpenROAD GUI requires X11 display forwarding from Windows host | Known | Start VcXsrv with "Disable access control" before launching ORFS container |
| SDC file is a minimal stub (no create_clock) | Known — planned for Phase 3 | Clock period is set via CLOCK_PERIOD in config.mk |
| ORFS P&R timeout set to 1800 s | May be insufficient for large designs | Increase `ORFS_TIMEOUT` in `agent_v3_3.py` |
